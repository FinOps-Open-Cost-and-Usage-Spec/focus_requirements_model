Introduction

FOCUS Requirement Model – Analyzer Guidelines

Overview

The FOCUS Requirement Model is a structured representation of all normative requirements defined in the FinOps Open Cost and Usage Specification (FOCUS).
Each requirement has been converted into a machine-readable JSON expression to enable automated validation, analysis, and future modeling activities.

The Requirement Model Analyzer application provides FinOps Practitioners and FOCUS Members with an interactive way to explore, filter, and analyze these requirements directly from the model database.
It helps users understand how each rule applies, how it connects to others, and which dataset attributes or columns it governs.

!TIP You can access and explore the live analyzer via the top navigation menu or directly at
Access the Requirement Model Analyzer

Purpose

The Requirement Model Analyzer application serves three key goals:

  1. For FinOps Practitioners:
    To review the normative requirements of the FOCUS Specification and understand how each applies to datasets, columns, and attributes.
  2. For FOCUS Members and Contributors:
    To evaluate existing model content, validate alignment with the specification, and prepare new requirements that can later be modeled using the Requirement Modeler (currently available in reader mode).
  3. For All Users:
    To search, filter, and export normative requirements for use in documentation, conformance tests, or model review sessions.

Tool Structure

The Analyzer application consists of two primary components:

  1. Filter Area – used to refine your search and select requirement subsets.
  2. Analyzer Table – dynamically updates to display all matching requirements and their detailed properties.

Step-by-Step Guide

1. Opening the Analyzer

  • Click on the ANALYZER top menu.
  • Wait for the JSON model to load (the loading time may vary depending on your network).

2. Understanding the Requirement Structure

Each requirement is stored as a JSON object with the following core properties:

PropertyDescription
ModelRuleIdThe formal ID of the requirement within the model.
FunctionDefines the rule type. Valid values: Composite, Format, Nullability, Presence, Type, Validation.
ReferenceThe column or attribute this rule applies to.
EntityTypeIndicates whether the rule applies to a Column, Attribute, or the Dataset.
ModelVersionIntroducedSpecifies the FOCUS version where the rule was added.
StatusIndicates whether the requirement is Active, Deprecated, or Removed.
NotesShort, free-form notes from the model documentation.
ApplicabilityCriteriaConditions that must be true for the rule to apply.
TypeDistinguishes between Static and Dynamic rules. Static rules can be validated without external data. Dynamic rules needs further information to be validated.
ValidationCriteriaA nested object defining the normative requirement, condition, and dependencies.

3. Understanding the Validation Criteria Object

Each rule also contains a nested ValidationCriteria object:

PropertyDescription
MustSatisfyThe normative text describing the rule’s requirement.
KeywordThe BCP-14 keyword governing the rule: MUST, RECOMMENDED, SHOULD, MAY, or OPTIONAL.
RequirementDefines what is required to fulfill the normative intent.
ConditionDescribes the conditions under which the requirement applies.
DependenciesIdentifies related or prerequisite rules.

4. Using the Filter Area

The Filter Area allows you to refine requirements by any of the following properties:

  • Model Version
  • Requirement ID (MRID)
  • Function
  • Reference
  • Entity
  • Keyword
  • Type
  • Applicability Criteria
  • Requirement – CheckFunction
  • Condition – CheckFunction

Each filter shows the number of requirements meeting that condition.
For example, FOCUS version 1.2 contains 621 normative requirements.

You can apply multiple filters simultaneously to narrow the dataset.
The table updates automatically with each filter selection.

5. Working with the Analyzer Table

The Analyzer Table displays one row per requirement.
Each row includes:

  • ID – The model rule identifier.
  • Applicability Criteria – Conditions for rule application.
  • MustSatisfy – The normative statement or expected behavior.
  • Requirements & Conditions – The arguments and logical conditions.
  • Dependencies – Related rules or logical references.
  • Resources – Click in the YAML and Mermaid diagram representations for visualization.

6. Searching and Viewing Requirements

  • Use the Search Box to look up any property by typing part of its value.
  • Adjust Show Items to control how many results appear at once.
  • Click a row to view more detailed information in the Requirement Reader.

7. Exporting Data

You can export the content of your current view:

  • Click Export symbol → Ensure you can view on the table the desired selection.
  • The exported file will contain the same fields displayed in the Analyzer Table.
  • Use this feature to share filtered sets or conduct offline analysis.

8. Requirement Modeler - Reader (Beta)

The Requirement Modeler - Reader lets you open any single requirement for detailed inspection.
At present, it functions as a read-only viewer, displaying the JSON object and associated diagrams.
Future versions will add editing and modeling capabilities.

!NOTE The Reader tool is in beta. We welcome your feedback and suggestions —
please share them via the project’s issue tracker or contact form.

Feedback & Contributions

If you find errors, inconsistencies, or have suggestions for improving the Requirement Model or Analyzer or Modeler:

  • Open an issue in the focus-requirements folder on GitHub, or
  • Submit feedback via the Feedback Form

Your input helps ensure the accuracy and completeness of the FOCUS Requirements Model for all community members.

Overview

Requirements Model - Program for FOCUS

The FinOps Open Cost and Usage Specification (FOCUS) v1.2 defines normative requirements to ensure datasets are consistent, interoperable, and reliable.
To make these requirements actionable, we translate them into FOCUS Requirements Model — structured, programmatic rules that can be tested automatically.

This process turns written standards into machine-readable instructions, supporting conformance testing, certification, and faster adoption of the FOCUS specification.
